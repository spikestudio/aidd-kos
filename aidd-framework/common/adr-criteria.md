# ADR 判断基準（SSOT）

> このファイルは ADR（Architecture Decision Record）を作成すべき判断の基準を定義する唯一の正解源。
> `/aidd-create-adr`・`/aidd-epic-design` はこのファイルを参照する。

**判定優先順序:** 除外基準を推奨基準より先に確認する。除外基準に該当する場合は、推奨基準にも当てはまっていても ADR 化しない（例: ライブラリ選定であっても1 Feature 内の実装詳細であれば除外）。

---

## ADR 化を推奨する判断（検出基準）

以下に該当する判断は ADR 候補として記録する:

| 基準 | 例 |
|------|-----|
| 横断設計 | 複数 Feature に影響する技術判断 |
| 後変更困難 | DB スキーマ・外部 API 連携・通信プロトコル |
| 複数選択肢から選択 | フレームワーク選定・ライブラリ選定 |
| 業界標準からの逸脱 | カスタムフロー・独自プロトコル |
| 既存 ADR への影響 | 既存決定の変更・追加 |
| 新規 Bounded Context 追加 | ドメイン設計の新規境界 |
| NFR 達成のための非標準アプローチ | 性能・セキュリティ戦略の判断 |

---

## ADR にしない判断（除外基準）

以下に該当する場合は ADR を作成しない。Feature Design や実装フェーズで決める内容であり、ADR に記録しても価値が低い:

| 除外基準 | 具体例 | 代替記録先 |
|---------|-------|-----------|
| **実装詳細** | 特定関数の書き方・DDD パターンの適用方法・型の選択 | 実装コメント / Feature Design |
| **業界標準・FW 推奨に従うだけの選択** | 公式ドキュメントに記載された推奨パターンをそのまま採用 | 記録不要 |
| **Feature Design の範囲で決まる実装パターン** | 特定 Feature 内の Repository vs Handler の責務分担 | Feature Design |
| **PRD・Epic Plan に既に記録済みの決定** | 重複記録は ADR の価値を下げる | 既存ドキュメントを参照 |
| **1 Feature にのみ影響する判断** | 他の Feature・コンポーネントへの波及がない局所的判断 | Feature Design |

NEVER: 除外基準に該当する判断を ADR として作成する。ADR は「将来の実装者が『なぜこうなっているか』を知りたい横断的・不可逆的判断」のみを対象とする。
