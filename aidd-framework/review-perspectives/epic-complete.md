# epic-complete レビュー観点

`/aidd-review` が epic-complete（Epic完了）と判定したときに読み込まれる観点集。

**判定したいこと:** Epic が完成し、本番に出せる品質か？

**観点数:** 12

---

## 共通出力フォーマット

MANDATORY: 各観点の Agent は以下フォーマットで結果を返すこと。フォーマット逸脱時は orchestrator が再要求する。

```markdown
## 観点 N レビュー結果: [観点名]

### Critical
- [問題タイトル]
  - 該当箇所: [ファイル名:行 / AC-ID / PR 番号]
  - 理由: [なぜ問題か・観測根拠を 1〜2 行]
  - 修正方針: [具体的アクション]

### Suggestions
- [改善提案タイトル]
  - 該当箇所: [ファイル名:行]
  - 提案理由: [改善で得られる効果]

### OK Points
- [評価する設計判断]

### Analysis Notes
- レビュー限界: [確認できなかった範囲]
- 不確実性: [前提条件・情報不足箇所]
```

DO NOT: 各セクションを省略する。該当なしの場合は「- なし」と明記。
DO NOT: 推測のみでの指摘（観測根拠を必ず添える）。
MANDATORY: 該当箇所はファイル名 + 行番号（または AC-ID / PR 番号）まで具体化する。

## 観点 1: AC カバレッジ・受け入れ判断 — aidd-product-owner

### レビュー対象

- `docs/epics/[N]-context.md`（Epic コンテキスト）
- `docs/spec/[domain].md`（Epic Spec）
- 対応 PR の実装コード差分
- テストコード（特に E2E・結合）
- `task check:ac-coverage` の実行結果（AC カバレッジ）

### MANDATORY チェック項目

1. **AC ↔ テストのトレーサビリティ**
   - 全 AC に対応する E2E または結合テストが最低 1 件存在
   - テスト名 / describe ブロック / コメントに該当 AC-ID が記載されている
   - DO NOT: テスト名から AC-ID が逆引きできない

2. **AC ↔ 実装のトレーサビリティ**
   - 全 AC に対応する実装コード（関数・コンポーネント・エンドポイント）が存在
   - PR 内で実装漏れの AC がない

3. **受け入れ判断（UAT 視点）**
   - 実装機能を手動操作した場合に AC の Given/When/Then を全て満たす
   - ユーザー価値（So that）が観察可能（KPI / 操作時間短縮 / エラー削減 等）に実現されている

4. **エッジケース・例外系の実装**
   - 異常系 AC（バリデーション・認可エラー・外部エラー）に対応する実装が存在
   - 境界値・エッジケースの実装が AC の Then 句通りに動作する

5. **Definition of Done**
   - Epic DoD（テスト・ドキュメント・レビュー）が全て満たされている
   - DO NOT: 部分実装・暫定実装（モック・スタブ）が本番ブランチに残存

### 重要度判定基準

- **Critical**: 主要 AC の実装/テスト欠落・UAT で AC を満たさない・暫定実装の本番混入
- **Suggestion**: テスト追加・エッジケース強化
- **OK Point**: 全 AC が実装・テスト・UAT で受け入れ可能

## 観点 2: 業務フロー整合・トレーサビリティ — aidd-business-analyst

### レビュー対象

- `docs/epics/[N]-context.md`（Epic コンテキスト）
- `docs/spec/[domain].md`（Epic Spec）
- `docs/features/[N]-[slug]/design.md`（Feature 設計）
- 対応 PR の実装コード

### MANDATORY チェック項目

1. **業務フロー実装の一致**
   - Epic Spec（`docs/spec/`）の業務フロー（トリガー・アクション・結果）と実装の関数フローが対応する
   - 業務イベント間の前後関係（A 完了 → B 開始）が実装で守られている

2. **例外フローの実装**
   - 業務上の例外（差戻し・取消・タイムアウト・補償処理）が実装で扱われている
   - 業務的失敗時の状態遷移が「業務の言葉」（差戻し済み・取消済み 等）で扱われている
   - DO NOT: 業務的失敗を `Error: ...` 等のシステムエラーで終了させる

3. **業務ルールの実装**
   - `docs/spec/` の業務ルール表と実装が 1:1 マッピングできる（ルール ID で双方向トレース可能）
   - ルール間の優先順位・依存関係が実装で表現されている

4. **ステークホルダー視点の網羅**
   - 直接ユーザー以外（承認者・管理者）の業務操作が画面 or API で操作可能に実装されている
   - 各ステークホルダーの「実現したい / 避けたい」が実装結果に反映されている

5. **要件 ↔ 実装のトレーサビリティ**
   - `docs/spec/` の要件 ID と実装ファイルの対応が PR description または `docs/features/` に記録されている
   - 実装で省略した要件は別 Issue 化されている（後回し・将来検討）

### 重要度判定基準

- **Critical**: 業務フローの誤実装・業務ルールの誤適用・例外実装欠落
- **Suggestion**: 業務メッセージの精緻化・補償処理の追加検討
- **OK Point**: 業務とシステムの境界が美しく整合・トレーサビリティ完備

## 観点 3: ドメイン一貫性 — aidd-domain-expert

### レビュー対象

- 実装コード全般（PR 差分）
- `docs/glossary.md`
- ドメインモデル（`docs/features/[N]-design.md` 内）
- DB スキーマ・API レスポンス命名

### MANDATORY チェック項目

1. **用語の一貫性**
   - コードのクラス名・関数名・変数名が `docs/glossary.md` の用語と一致
   - DB テーブル名・カラム名が glossary の用語と一致（snake_case 等の規約は除く）
   - API のエンドポイントパス・パラメータ名が glossary と一致
   - UI のラベル・エラーメッセージが glossary と一致
   - DO NOT: glossary 未登録の独自用語をコードに導入

2. **ドメインモデルの実装整合**
   - 1 集約 = 1 トランザクション境界として実装されている
   - 他集約への参照は ID（外部キー / 識別子）のみで、直接オブジェクト参照を持たない
   - 不変条件（Invariant）が集約のコンストラクタ / メソッドで強制されている

3. **業界規制・法令の実装**
   - 業界規制（証跡保存期間 / 本人確認 / 同意取得タイミング）が Epic Spec（`docs/spec/`）の通りに実装されている
   - データ保護要件（保存時暗号化 / 通信時暗号化 / アクセスログ）が業界水準で実装されている

4. **業界固有のエッジケースの扱い（暗黙ルール由来）**
   - 業界特有のエッジケース（業務日 / 締日 / 年度跨ぎ / うるう年 / タイムゾーン）が実装でハンドリングされている
   - DO NOT: プロジェクト固有の業務例外（差戻し・取消・タイムアウト・補償処理）はここで重複指摘しない（観点 2「例外フローの実装」が責務）

5. **業界ベストプラクティス**
   - 業界標準のデータ形式・プロトコル（ISO 4217 通貨コード / ISO 8601 日時 / E.164 電話番号 等）が採用されている
   - DO NOT: 業界の慣用に反する命名・処理（例: 通貨を浮動小数で扱う 等）

### 重要度判定基準

- **Critical**: 業界規制違反・用語の誤用が広範囲・業界エッジケース未実装・通貨を float で扱う等
- **Suggestion**: 業界ベストプラクティスの段階的適用
- **OK Point**: 業界の語彙・慣行と完全合致した実装

## 観点 4: 設計整合（6 階層トレーサビリティ） — aidd-architect

### レビュー対象

- Epic Spec（`docs/spec/`）・`docs/business-context/`・Feature 設計（`docs/features/`）・実装コード（全層）
- `docs/architecture/adr/*.md`
- マスタドキュメント（規約・技術スタック）

### MANDATORY チェック項目

1. **6 階層トレーサビリティ**
   - Phase → Epic → Story → Feature → AC → Task → 実装 のトレースが全層で繋がる
   - 各層のドキュメントが上位層を引用または参照リンクで結ばれている
   - DO NOT: 中間層が抜けて Phase ↔ Task が直接結ばれる

2. **ADR への準拠**
   - 関連 ADR の決定事項が実装で守られている
   - ADR から逸脱している箇所には「ADR 更新済み」または「修正中」のコメントが存在

3. **設計のドメインモデル整合**
   - Feature 設計のドメインモデルと実装の集約・エンティティ・値オブジェクトが対応する
   - 集約境界・エンティティ属性が DB スキーマ・API レスポンスと整合する

4. **コンポーネント境界**
   - レイヤー間の依存方向が presentation → application → domain → infrastructure の一方向
   - DO NOT: domain 層が infrastructure に依存（依存逆転原則違反）

5. **横断的設計の整合**
   - 同種処理パターン（バリデーション・例外ハンドリング・ロギング・認可チェック）が Feature 間で統一されている
   - DO NOT: 同じ業務を複数の異なるパターンで実装

### 重要度判定基準

- **Critical**: ADR 違反・トレーサビリティ断絶・レイヤー依存逆転・同種処理の不統一が広範囲
- **Suggestion**: パターン統一・抽象化レベルの調整
- **OK Point**: 6 階層が美しく整合・横断パターンが統一されている

## 観点 5: コード品質 — skill:review + aidd-builder

このレビュー観点は 2 段階で処理する。

**段階 A（skill:review）**: PR 全体のコード品質レビューをランタイムのデファクトスタンダードスキルに委譲する。
**段階 B（aidd-builder）**: aidd 固有の規約準拠のみ追加確認する。

> `review` スキルは常に最新のレビュー品質を提供する。可読性・エラーハンドリング・重複コード・バグ検出は委譲して再実装しない。
> `epic-complete` の段階では必ず PR が存在するため、diff ではなく PR 全体を見る `review` スキルが適切。

### 段階 A: skill:review への委譲

`review` スキルを呼び出し、その結果をそのまま観点 5 の Critical / Suggestions として取り込む。

委譲内容（review スキルが担う）:
- PR 全体の正確性・バグ検出
- 可読性（命名・責務分離・ネスト深度）
- エラーハンドリング（例外握り潰し・伝播方針）
- 重複・デッドコード
- コメント・docstring 品質

### 段階 B: aidd 固有チェック — aidd-builder

段階 A に加え、以下の aidd 固有項目のみを確認する。

### MANDATORY チェック項目

1. **aidd 規約準拠**
   - `CLAUDE.md` / `AGENTS.md` の全規約に違反する箇所がない
   - リンター警告が 0 件
   - 型チェッカーエラーが 0 件
   - DO NOT: `// eslint-disable` `@ts-ignore` `@ts-expect-error` 等の suppress を追加（既存承認済みを除く）

### 重要度判定基準

- **Critical**: 型エラー残存・suppress 追加（承認なし）・CLAUDE.md 規約の重大違反
- **Suggestion**: 規約軽微違反・lint 警告の解消
- **OK Point**: 規約完全準拠・型エラー 0

## 観点 6: テスト品質 — aidd-qa-engineer

### レビュー対象

- 対応 PR のテストコード
- カバレッジレポート（存在する場合）
- `CLAUDE.md` / `AGENTS.md` のプロジェクト固有ルール

### MANDATORY チェック項目

1. **AC ↔ テストのマッピング**
   - 全 AC に対応するテストが存在し、テスト名 / describe に AC-ID が記載
   - テストが「AC を検証」する意図で書かれている（実装の挙動を後追いするだけのテストではない）

2. **テスト粒度・テストピラミッド**
   - テスト件数比率: ユニット 70% / 結合 20% / E2E 10% の目安に近い
   - DO NOT: E2E が 50% を超える（過剰 E2E）または unit が 0 件

3. **エッジケース・境界値テスト**
   - 異常系 AC に対応する異常系テストが存在
   - 境界値テスト（最小値・最大値・空入力・最大長入力）が該当 AC で存在
   - エラーパスのテスト件数が正常系の 30% 以上

4. **テスト独立性・再現性**
   - テストが任意の順序で実行可能（順序依存がない）
   - 外部依存（DB / 外部 API / 時刻 / ランダム）が以下のいずれかで置換されている:
     - DB: testcontainers / インメモリ DB
     - 外部 API: msw / jest.mock / Pact
     - 時刻: clock mock / fake timer
     - ランダム: seed 固定
   - DO NOT: 同じテストが連続実行で時々失敗する（フレーキー）

5. **テストデータ・フィクスチャ**
   - テストデータが意図を反映する命名（`validUserWithAdminRole` 等）
   - DO NOT: マジックナンバー・マジック文字列（`"foo"` `1234` 等）が意図不明で散在
   - フィクスチャ / ファクトリが共通化されている（重複ボイラープレートが 5 箇所以上ない）

### 重要度判定基準

- **Critical**: AC 未テスト・E2E 0 件・フレーキーテスト存在・外部依存の本物呼び出し
- **Suggestion**: テストデータ整理・粒度バランス改善
- **OK Point**: テスト戦略が機能・カバレッジバランスが良い

## 観点 7: 運用・観測準備 — aidd-sre

### レビュー対象

- 対応 PR の実装コード（ログ・メトリクス・トレース埋め込み箇所）
- `CLAUDE.md` / `AGENTS.md` のプロジェクト固有ルール
- アラート設定・ダッシュボード（存在する場合）

### MANDATORY チェック項目

1. **ログの完全性**
   - 重要処理（外部 API 呼出・DB 更新・状態遷移）の開始・終了がログ出力されている
   - エラー発生箇所が ERROR レベルでログ出力されている
   - ログレベル使い分け: DEBUG=調査時のみ / INFO=正常フロー / WARN=回復可能異常 / ERROR=要対応
   - 構造化ログ（JSON）で `request_id` `trace_id` 付与
   - DO NOT: PII（メールアドレス・電話番号・氏名）がログに含まれる

2. **メトリクス**
   - SLI（レイテンシ・エラー率・スループット）に必要なメトリクスが取得できる埋め込みがある
   - ビジネスメトリクス（重要操作の成功回数・失敗回数）も取得可能

3. **トレース**
   - リクエスト ID / トランザクション ID で処理を追跡可能
   - 外部 API 呼び出し・DB クエリ・非同期ジョブのトレースが取れる

4. **エラーハンドリング戦略**
   - 外部依存呼び出しにタイムアウト設定が存在（明示的にミリ秒指定）
   - リトライ・サーキットブレーカー・フォールバックの戦略が必要箇所に実装されている
   - ユーザーへのエラー表示と内部エラー記録が分離（ユーザー向けは技術詳細を含まない）

5. **運用ドキュメント**
   - ランブック（起動・停止・障害対応）が `docs/playbook/` または同等場所に存在
   - オンコール手順・エスカレーション基準が記載されている

### 重要度判定基準

- **Critical**: 主要処理のログなし・タイムアウト未設定・PII がログに含まれる・ランブック皆無
- **Suggestion**: メトリクス追加・トレース粒度改善
- **OK Point**: 観測性が設計から組み込まれている

## 観点 8: パフォーマンス — aidd-performance-engineer

### レビュー対象

- 対応 PR の実装コード（DB クエリ・API 呼び出し・データ取得パターン）
- DB スキーマ・インデックス設定
- 負荷テスト結果（存在する場合）
- NFR（性能要件）

### MANDATORY チェック項目

1. **N+1 クエリ**
   - ループ（for / map / forEach）内で DB クエリや API 呼び出しが発生していない
   - ORM の lazy loading でループ内アクセス時に N+1 が発生していない
   - 必要な関連データが eager loading / JOIN / batch get で取得されている

2. **インデックス**
   - 頻繁にクエリされるカラム（WHERE 句・JOIN 句・ORDER BY 句）にインデックスが張られている
   - 複合インデックスの列順がクエリの選択性に従っている
   - DO NOT: 使われていないインデックスが残存

3. **キャッシュ戦略**
   - 高頻度参照データ（マスタデータ・セッション・認可情報）がキャッシュされている
   - キャッシュ失効戦略（TTL 値 + イベント駆動の組み合わせ）が定義されている
   - キャッシュ層（インメモリ / Redis / CDN）の使い分けが用途に合致

4. **大量データ・スケーラビリティ**
   - 一覧取得 API にページネーション（offset または cursor）が実装されている
   - 同時アクセス時のロック範囲が最小化されている（行ロック優先）
   - 重い処理（バッチ・集計・通知送信）が非同期化されている

5. **性能要件達成**
   - NFR の性能目標（レイテンシ・スループット）を満たすベンチマーク結果が存在
   - 負荷テスト結果がベースライン（前 Phase or 既存）を 10% 以上劣化させていない

### 重要度判定基準

- **Critical**: 明らかな N+1・主要クエリの全件スキャン・性能要件未達
- **Suggestion**: 早すぎる最適化を避けつつ将来のボトルネック候補
- **OK Point**: 計測に基づいた最適化が施されている

## 観点 9: セキュリティレビュー — aidd-security-specialist

### レビュー対象

- 対応 PR の実装コード（認証・認可・入力処理・出力処理）
- 依存ライブラリ（`package.json` / `pyproject.toml` 等）
- 監査ログ実装
- secrets / environment 変数の取扱い

### MANDATORY チェック項目

1. **OWASP Top 10 リスク**
   - SQL Injection: 全 SQL がパラメータ化クエリ（ORM 経由 or prepared statement）
   - XSS: ユーザー入力の表示で HTML エスケープが行われている
   - CSRF: 状態変更 API に CSRF トークン検証が存在
   - SSRF: 外部 URL アクセスに allowlist 検証が存在
   - 認可バイパス: IDOR（直接オブジェクト参照）の防止策が存在

2. **入力バリデーション**
   - 全ての外部入力（フォーム / API / URL パラメータ）が型・長さ・形式・範囲で検証される
   - 検証は allowlist 方式（許可パターンを明示）を基本とする

3. **認可チェック**
   - 各エンドポイント・各操作で認可チェックが実行される
   - リソースレベルの認可（自分のデータか・テナント分離）が機能している
   - DO NOT: 認可チェックが abundant（過剰）ではなく一元化されたミドルウェア or デコレータで実装

4. **依存ライブラリの脆弱性**
   - `npm audit` / `pip audit` / 同等コマンドで High 以上の脆弱性が 0 件
   - lock file が存在しコミットされている
   - Dependabot / Renovate の自動更新 PR が機能している

5. **秘密情報管理**
   - secrets がコードハードコードされていない（grep で `password=` `api_key=` `secret=` 等が hit しない）
   - secrets は環境変数または KMS / Vault 経由で取得される
   - ログ・エラーメッセージに secrets / PII が含まれない

6. **暗号化・通信**
   - 保存時および転送時の暗号化（マネージドキー or CMK、TLS 1.2 以上）
   - HTTP セキュリティヘッダー（CSP / HSTS（max-age ≥ 31536000）/ X-Content-Type-Options / X-Frame-Options / Referrer-Policy）が設定されている

7. **ログ・監査・監視**
   - ネットワークリレー（LB / API GW / CDN）のアクセスログが有効
   - アプリケーションログが集中管理されている（タイムスタンプ / 相関 ID / レベル付き）
   - 認証失敗・権限昇格に対するアラートが設定されている
   - ログ保持期間が最低 90 日
   - 重要なデータ変更の監査ログが存在する

8. **権限・ネットワーク制限**
   - 最小権限（ワイルドカードのリソース・アクション禁止、読み取り/書き込みロール分離）
   - ネットワークはデフォルト拒否、`0.0.0.0/0` インバウンドは公開 LB のポート 80/443 のみ、プライベートサブネットは NAT 経由
   - アプリレベルのアクセス制御がデフォルト拒否、明示的な CORS オリジン、全リクエストで JWT 検証

9. **ハードニング・完全性・フェイルセーフ**
   - デフォルト認証情報禁止、本番エラーにスタックトレースを含めない、未使用機能を削除
   - 安全なデシリアライズ、成果物のチェックサム / 署名
   - 全ての外部呼び出しに明示的なエラーハンドリング、フェイルクローズ、ユーザー向けは汎用エラーメッセージ

10. **認証・MFA・サプライチェーン**
    - パスワード最低 8 文字 + 漏洩 DB チェック、適応型ハッシュ、MFA（管理者は必須）
    - 依存関係をピン留め / ロックファイル使用、脆弱性スキャナー、SBOM、CI/CD での `latest` タグ禁止
    - セキュアな設計（認証 / 認可 / 決済 = 専用モジュール）、レート制限

### 重要度判定基準

- **Critical**: OWASP リスクの存在・認可漏れ（IDOR）・High 以上の CVE 含有・secrets ハードコード・TLS 未対応・MFA 未実装（管理者）・本番スタックトレース露出
- **Suggestion**: 多層防御の追加・監査強化
- **OK Point**: 脅威モデルに基づく対策が網羅的・一元化されている
