# 技術スタック推定ガイド（共通骨格）

MANDATORY: prototype 系サブスキル（`/aidd-prototype-api`, `/aidd-prototype-cli` 等）が技術スタックを推定する際の共通手順。

スキル固有の `tech_stack` キー・判定ルールテーブル・フレームワーク別慣習は、各サブスキルの `SKILL.md` 「## 技術スタック推定差分」セクションで定義する。

---

## 推定の優先順序

1. **CLAUDE.md の `tech_stack` セクション** — 最も信頼性が高い
2. **プロジェクトファイルの分析** — コード構造から推定
3. **機能意図・プロジェクト名のキーワード** — 最終手段

---

## 共通推定手順

### Step 1: CLAUDE.md を確認

```
docs/CLAUDE.md または .claude/CLAUDE.md に以下が存在するか:
  tech_stack:
    {{TECH_STACK_KEY}}: [スキル固有の候補]
```

存在すれば、その値を採用し Step 2/3 をスキップする。

### Step 2: プロジェクトファイルを確認

スキル固有の判定ルールテーブル（`SKILL.md` 「## 技術スタック推定差分」の `DETECTION_RULES`）に従い、`package.json` / `go.mod` / `pyproject.toml` / `Cargo.toml` / `pom.xml` 等の依存定義を分析する。

判定がつかない場合は Step 3 へ。

### Step 3: 機能意図・プロジェクト名のキーワード

スキル固有のキーワードマップ（`SKILL.md` 「## 技術スタック推定差分」の `KEYWORD_MAP`）に従いキーワードベースで推定する。

それでも判定不能なら、デフォルト（スキル固有・`DEFAULT`）を採用する。

---

## 推定結果の提示フォーマット

```
技術スタック推定結果:
  根拠: [Step 1/2/3 のどこで判定したか、具体的な検出箇所]
  推定: [候補名]
  慣習: [そのフレームワーク固有の設計慣習の概要（スキル側で定義）]
  確認: この技術スタックでよいですか？（違う場合は指定してください）
```

MANDATORY: ユーザーに必ず確認する。承認なしに次ステップへ進まない。

---

## スキル側で定義する差分

サブスキルは自身の `SKILL.md` 「## 技術スタック推定差分」セクションで以下を定義する:

| スロット | 内容 | 必須 |
|---------|------|------|
| `TECH_STACK_KEY` | CLAUDE.md `tech_stack` の参照キー名（例: `api`, `cli_framework`） | ✅ |
| `DETECTION_RULES` | プロジェクトファイル判定ルールテーブル（依存定義 → 候補） | ✅ |
| `KEYWORD_MAP` | 機能意図・プロジェクト名のキーワード判定テーブル | 任意 |
| `DEFAULT` | 判定不能時のデフォルト候補 | ✅ |
| `FRAMEWORK_CONVENTIONS` | フレームワーク別の設計慣習（候補ごとの特徴・コード例） | 任意 |
